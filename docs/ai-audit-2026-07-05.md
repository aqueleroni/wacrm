# PARTE 1 — SEGURANÇA (corrigir antes de qualquer outra coisa)

## S1 + S2 + S3 · RPCs de IA abertas para o mundo 🔴 ✅ CORRIGIDO (migration 035)

**Onde:**
- `supabase/migrations/029_ai_reply.sql:118-131` (`claim_ai_reply_slot`)
- `supabase/migrations/030_ai_knowledge.sql:155-204` (`match_ai_knowledge_fts`, `match_ai_knowledge_semantic`)
- `supabase/migrations/032_ai_agent_intelligence.sql:123-143` (`match_ai_agent_memory_fts`)

**O problema (confirmado no banco ao vivo pelos advisors do Supabase):**

1. **`claim_ai_reply_slot`** foi criada `SECURITY DEFINER` mas a migration 029 **esqueceu o `REVOKE`/`GRANT`** que a 030 aplica (a própria 030 comenta: *"would otherwise default to PUBLIC"*). Resultado: qualquer pessoa — **até sem login**, via `POST /rest/v1/rpc/claim_ai_reply_slot` — pode passar qualquer `conversation_id` e incrementar `ai_reply_count` até o teto. Efeito: **silenciar permanentemente o auto-reply de qualquer conversa de qualquer conta** (negação de serviço direcionada e invisível).

2. As três **`match_*`** têm `REVOKE FROM PUBLIC` nas migrations, mas o advisor confirma que no projeto ao vivo elas estão executáveis por `anon` **e** `authenticated`. Como são `SECURITY DEFINER` (ignoram RLS) e o único filtro é o `p_account_id` **passado pelo chamador**, qualquer visitante anônimo pode enumerar UUIDs de conta e **ler a base de conhecimento e a memória interna de qualquer tenant** — incluindo preços, políticas, fatos sobre clientes. Mesmo que o `REVOKE` de `anon` seja reaplicado, todo usuário logado de *outra* conta continua podendo ler, porque nenhuma função verifica filiação.

**Por que é crítico:** é vazamento de dados entre contas (multi-tenant desde a migration 017) somado a DoS do agente. Não depende de bug no app — é a API REST do Supabase exposta diretamente.

**Correção — nova migration `035_ai_rpc_hardening.sql`:**

```sql
-- ============================================================
-- 035_ai_rpc_hardening.sql — trava as RPCs de IA
-- ============================================================

-- 1) claim_ai_reply_slot: só o bot (service_role) precisa chamar.
REVOKE ALL ON FUNCTION public.claim_ai_reply_slot(uuid, integer) FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public.claim_ai_reply_slot(uuid, integer) TO service_role;

-- 2) match_*: defesa em profundidade — além de revogar anon,
--    a própria função verifica filiação. service_role tem
--    auth.uid() NULL e passa; usuário logado só passa se for
--    membro da conta pedida.
CREATE OR REPLACE FUNCTION public.match_ai_knowledge_fts(
  p_account_id uuid, p_query text, p_match_count integer
)
RETURNS TABLE (id uuid, content text, rank real) AS $$
  SELECT c.id, c.content,
         ts_rank(c.fts, plainto_tsquery('simple', p_query)) AS rank
  FROM ai_knowledge_chunks c
  WHERE c.account_id = p_account_id
    AND (auth.uid() IS NULL OR is_account_member(p_account_id))
    AND c.fts @@ plainto_tsquery('simple', p_query)
  ORDER BY rank DESC
  LIMIT GREATEST(p_match_count, 0);
$$ LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public;

-- (repetir o mesmo guard em match_ai_knowledge_semantic e
--  match_ai_agent_memory_fts)

REVOKE ALL ON FUNCTION public.match_ai_knowledge_fts(uuid, text, integer) FROM PUBLIC, anon;
GRANT EXECUTE ON FUNCTION public.match_ai_knowledge_fts(uuid, text, integer) TO authenticated, service_role;
-- idem para as outras duas
```

> ⚠️ Nota: `anon` ter EXECUTE mesmo com `REVOKE FROM PUBLIC` nas migrations indica que os *default privileges* do schema re-concederam, ou que a ordem de aplicação via MCP pulou os REVOKEs. Depois da migration, **rode o advisor de novo** para confirmar que sumiu. O guard `auth.uid() IS NULL OR is_account_member(...)` protege mesmo se os grants regredirem no futuro.

## S4 · Outras `SECURITY DEFINER` expostas ao `anon` 🟠 ✅ CORRIGIDO (migration 035)

**Onde:** banco ao vivo (advisors). Funções como `_bcast_bump` (corrompe contadores de broadcast de qualquer conta), `recompute_broadcast_counts`, `record_webhook_failure` (desativa endpoints de webhook alheios após N chamadas), `merge_duplicate_contacts`, além de funções de trigger (`handle_new_user`, `broadcast_recipient_aggregate_trigger`, `notify_conversation_assigned`) que nunca deveriam ser chamáveis via REST.

**Por que importa:** cada uma é um vetor menor, mas `_bcast_bump(bid, col, delta)` sem escopo permite adulterar métricas de qualquer broadcast, e `record_webhook_failure` permite desligar a integração de webhook de outra conta.

**Correção:** na mesma migration 035:

```sql
-- Funções de trigger: ninguém chama via RPC.
REVOKE ALL ON FUNCTION public.handle_new_user() FROM PUBLIC, anon, authenticated;
REVOKE ALL ON FUNCTION public.broadcast_recipient_aggregate_trigger() FROM PUBLIC, anon, authenticated;
REVOKE ALL ON FUNCTION public.notify_conversation_assigned() FROM PUBLIC, anon, authenticated;

-- Internas de engine: só service_role.
REVOKE ALL ON FUNCTION public._bcast_bump(uuid, text, integer) FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public._bcast_bump(uuid, text, integer) TO service_role;
REVOKE ALL ON FUNCTION public.record_webhook_failure(uuid, integer) FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public.record_webhook_failure(uuid, integer) TO service_role;
REVOKE ALL ON FUNCTION public.recompute_broadcast_counts(uuid) FROM PUBLIC, anon;
REVOKE ALL ON FUNCTION public.merge_duplicate_contacts() FROM PUBLIC, anon;

-- As de convite/membro (peek_invitation, redeem_invitation, set_member_role,
-- remove_account_member, transfer_account_ownership, touch_presence,
-- is_account_member) validam auth.uid() internamente, mas revogue anon
-- onde o fluxo não exige anônimo (só peek_invitation precisa de anon).
```

## S5 · Bucket `chat-media` público e listável 🟠

**Onde:** Storage — buckets `chat-media`, `avatars`, `account-branding`, `flow-media` com política SELECT ampla em `storage.objects`.

**Por que importa:** `chat-media` guarda **fotos, áudios e documentos de conversas de clientes no WhatsApp**. Público + listável significa que qualquer pessoa pode enumerar e baixar toda a mídia de todos os tenants. Avatares/branding serem públicos é aceitável; mídia de chat não.

**Correção:** remover a política SELECT ampla dos buckets (URLs públicas continuam funcionando sem ela, sem permitir listagem). Para `chat-media`, o ideal é migrar para bucket **privado + signed URLs** geradas na API (`createSignedUrl(path, 3600)`) — o inbox já roda server-side e pode assinar na hora de renderizar.

## B5 · Ciphertext da API key legível por qualquer membro 🟡

**Onde:** `supabase/migrations/029_ai_reply.sql:65-67` — policy `ai_configs_select` dá SELECT em **todas as colunas** para qualquer membro (viewer+). A rota `/api/ai/config` remove `api_key` da resposta, mas um viewer com supabase-js consegue `select('api_key, embeddings_api_key')` direto.

**Por que importa:** é só o ciphertext (AES-256-GCM), então sem a `ENCRYPTION_KEY` não vira plaintext — mas viola defesa em profundidade: se a chave de criptografia vazar um dia, todo histórico de leitura vira exposição.

**Correção:** column privileges ou uma view:

```sql
REVOKE SELECT ON public.ai_configs FROM authenticated;
GRANT SELECT (id, account_id, provider, model, system_prompt, conversation_examples,
              is_active, auto_reply_enabled, auto_reply_max_per_conversation,
              memory_auto_extract, created_at, updated_at)
  ON public.ai_configs TO authenticated;
```
(As rotas server-side que precisam da chave usam service_role ou o SSR client de admin — verificar que `loadAiConfig` no caminho de draft roda com um client que mantenha acesso; se necessário, mover a leitura da chave para o admin client.)

## B2 · Uma automação `keyword_match` ativa silencia a IA da conta inteira 🔴

**Onde:** `src/lib/ai/auto-reply.ts:60-67`

```ts
const { data: autoResponders } = await db
  .from('automations')...
  .in('trigger_type', ['new_message_received', 'keyword_match'])
  .limit(1)
if (autoResponders && autoResponders.length > 0) return  // ← IA desiste SEMPRE
```

**O problema:** o gate verifica se **existe** automação ativa desses tipos — não se ela **respondeu esta mensagem**. Uma conta que cria uma única automação de keyword (ex.: "promoção" → envia cupom) perde o auto-reply de IA em **100% das mensagens**, inclusive as que não contêm a keyword. O comentário no código admite que é para evitar double-texting, mas o custo é desproporcional: é o bug que mais reduz a utilidade real do agente.

**Correção:** avaliar se alguma automação de fato vai responder, reaproveitando a mesma lógica de matching do engine:

```ts
// auto-reply.ts — substituir o gate existencial por um gate de match real
import { keywordMatches } from '@/lib/automations/engine' // extrair helper puro

const { data: autoResponders } = await db
  .from('automations')
  .select('trigger_type, trigger_config')
  .eq('account_id', accountId)
  .eq('is_active', true)
  .in('trigger_type', ['new_message_received', 'keyword_match'])

const willAutoRespond = (autoResponders ?? []).some((a) =>
  a.trigger_type === 'new_message_received' ||
  keywordMatches(a.trigger_config, inboundText),
)
if (willAutoRespond) return
```
Isso exige passar `inboundText` no `DispatchArgs` (o webhook já o tem) e exportar do engine o predicado puro de keyword — que já existe lá dentro para decidir a execução. Assim `new_message_received` (responde tudo) continua vencendo a IA, e `keyword_match` só vence quando bate.

## B3 · Starvation no cron de extração de memória 🟠

**Onde:** `src/lib/ai/memory-extract.ts:194-208` (`findConversationsForExtract`)

**O problema:** a query busca as `limit * 3` conversas **mais antigas** (`order updated_at asc`) e filtra *em JavaScript* as já extraídas. Conversas antigas já extraídas e sem atividade nova **continuam ocupando a janela de busca para sempre**. Com apenas ~60 conversas dormentes no banco, o filtro descarta a janela inteira e o cron passa a processar **zero** conversas — a extração automática de memória morre silenciosamente (o pior tipo de falha: `{processed: 0}` parece "nada a fazer").

O índice parcial `conversations_memory_extract_idx ... WHERE ai_memory_extracted_at IS NULL` (migration 033) nem é usado, porque a query não filtra por essa coluna.

**Correção:** filtrar no SQL com uma RPC (PostgREST não compara coluna-com-coluna):

```sql
-- migration 036
CREATE OR REPLACE FUNCTION public.find_conversations_for_memory_extract(
  p_cutoff timestamptz, p_limit integer
)
RETURNS TABLE (id uuid, account_id uuid, contact_id uuid) AS $$
  SELECT c.id, c.account_id, c.contact_id
  FROM conversations c
  JOIN ai_configs cfg ON cfg.account_id = c.account_id
    AND cfg.memory_auto_extract AND cfg.is_active
  WHERE c.updated_at < p_cutoff
    AND (c.ai_memory_extracted_at IS NULL
         OR c.updated_at > c.ai_memory_extracted_at)
  ORDER BY c.updated_at ASC
  LIMIT GREATEST(p_limit, 0);
$$ LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public;

REVOKE ALL ON FUNCTION public.find_conversations_for_memory_extract(timestamptz, integer)
  FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public.find_conversations_for_memory_extract(timestamptz, integer)
  TO service_role;

-- índice que realmente serve a query:
DROP INDEX IF EXISTS conversations_memory_extract_idx;
CREATE INDEX conversations_memory_extract_idx
  ON conversations (updated_at)
  WHERE ai_memory_extracted_at IS NULL OR updated_at > ai_memory_extracted_at;
-- (predicado com coluna-vs-coluna é permitido em índice parcial)
```
E em `findConversationsForExtract`, trocar as duas queries por `db.rpc('find_conversations_for_memory_extract', {...})`.

## B4 · Webhook sem idempotência de mensagens inbound 🟠

**Onde:** `src/app/api/whatsapp/webhook/route.ts:668-683` (INSERT em `messages` sem checar `message_id` existente; `idx_messages_message_id` não é único).

**O problema:** a Meta **reenvia webhooks** quando não recebe 200 em tempo hábil (deploys, cold starts, timeouts de rede). Cada retry insere a mensagem de novo → mensagem duplicada no inbox, `unread_count` inflado, automações disparadas 2×, **e a IA responde 2×** (consumindo 2 slots do cap e gastando 2 chamadas de LLM). O comentário na linha 360 nota que Meta IDs não são globalmente únicos entre contas — mas são únicos *por conversa*.

**Correção:**

```sql
-- migration 036 (continuação) — dedupe por conversa
CREATE UNIQUE INDEX IF NOT EXISTS idx_messages_conv_wamid_unique
  ON messages (conversation_id, message_id)
  WHERE message_id IS NOT NULL AND sender_type = 'customer';
```
```ts
// webhook route — trocar .insert() por upsert ignorando duplicata
const { error: msgError, data: inserted } = await supabaseAdmin()
  .from('messages')
  .upsert({ ...row }, { onConflict: 'conversation_id,message_id', ignoreDuplicates: true })
  .select('id')
if (!inserted?.length) return // retry da Meta — já processado, não redisparar flows/IA
```

## B6 · Dedupe de memória quebra com volume 🟡

**Onde:** `src/lib/ai/memory-extract.ts:156-175` (`hasSimilarMemory`)

**O problema:** busca `limit(50)` **sem ordenação e sem filtro de contato** — o Postgres retorna 50 linhas arbitrárias. Passando de 50 memórias na conta, o dedupe passa a comparar contra um subconjunto aleatório e começa a inserir duplicatas pendentes, poluindo a fila de aprovação. Além disso são 1–3 roundtrips por conversa processada.

**Correção:** escopar por contato + conta, ordenar por recência, e comparar também as account-wide:

```ts
const { data } = await db
  .from('ai_agent_memory')
  .select('content')
  .eq('account_id', accountId)
  .or(`contact_id.is.null${contactId ? `,contact_id.eq.${contactId}` : ''}`)
  .in('status', ['pending', 'approved', 'rejected'])
  .order('updated_at', { ascending: false })
  .limit(100)
```
(Melhor ainda: usar a própria FTS — `match_ai_agent_memory_fts` com o candidato como query e um threshold de rank — em vez de substring bidirecional.)

## B7 · Race condition no `unread_count` 🟡

**Onde:** `src/app/api/whatsapp/webhook/route.ts:692` — `unread_count: (conversation.unread_count || 0) + 1` é read-modify-write; dois inbounds simultâneos perdem um incremento.

**Correção:** RPC atômica (mesmo padrão do `claim_ai_reply_slot`):

```sql
CREATE OR REPLACE FUNCTION public.bump_conversation_inbound(
  p_conversation_id uuid, p_last_text text
) RETURNS void AS $$
  UPDATE conversations
  SET unread_count = unread_count + 1,
      last_message_text = p_last_text,
      last_message_at = now(),
      updated_at = now()
  WHERE id = p_conversation_id;
$$ LANGUAGE sql SECURITY DEFINER SET search_path = public;
REVOKE ALL ON FUNCTION public.bump_conversation_inbound(uuid, text) FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public.bump_conversation_inbound(uuid, text) TO service_role;
```

# PARTE 3 — PERFORMANCE

## P1 · Gates sequenciais no auto-reply (latência antes do LLM)

**Onde:** `src/lib/ai/auto-reply.ts:49-81` — 4 roundtrips **em série** (config → automações → conversa → mensagens) antes de sequer montar o prompt.

**Correção:** paralelizar o que não depende entre si:

```ts
const [config, autoResponders, conv, messages] = await Promise.all([
  loadAiConfig(db, accountId),
  db.from('automations').select('trigger_type, trigger_config')
    .eq('account_id', accountId).eq('is_active', true)
    .in('trigger_type', ['new_message_received', 'keyword_match']),
  db.from('conversations').select('assigned_agent_id, ai_autoreply_disabled, ai_reply_count')
    .eq('id', conversationId).maybeSingle(),
  buildConversationContext(db, conversationId),
])
// gates na mesma ordem de antes, agora sem custo de latência somado
```
Economia típica: 150–400 ms por resposta (relevante: o cliente está esperando no WhatsApp).

## P2 · COUNT "existe KB/memória?" a cada turno

**Onde:** `knowledge.ts:98-106` e `memory.ts:40-48`. Duas queries extras por resposta só para descobrir se a conta tem conteúdo. **Correção:** cachear o booleano por conta em memória com TTL de ~60s (Map módulo-level, mesmo padrão do rate-limit), invalidando nas rotas de escrita de KB/memória. Em contas ativas isso elimina 2 roundtrips de cada resposta.

## P3 · Sem retry/backoff nos providers

**Onde:** `providers/openai.ts`, `providers/anthropic.ts`, `embeddings.ts` — uma falha 429/529/5xx transitória derruba o draft/auto-reply na hora. **Correção:** wrapper com 2 retries exponenciais (respeitando `Retry-After`) apenas para `rate_limited`/`provider_error` 5xx — nunca para `invalid_key`. Isso é o que os SDKs oficiais fazem por padrão.

## P4 · Consolidação de memória 100% serial

**Onde:** `memory-consolidate.ts:34-49` — até 20 conversas × (1 chamada LLM + 2-4 queries) em série dentro de uma invocação de cron. Com LLM a ~2-5s cada, estoura folga de timeout. **Correção:** lotes de 3-5 com `Promise.allSettled`, mantendo o try/catch por item.

## P5 · Rate limit em memória não funciona em serverless

**Onde:** `src/lib/rate-limit.ts` (documentado no próprio arquivo). Em Vercel/multi-instância cada lambda tem seu Map — o limite real vira `limite × nº de instâncias`, e os caps de custo de IA (`aiDraft`, `aiDraftAccount`) deixam de proteger a chave BYO. **Correção:** ao escalar além de 1 VPS, trocar `checkRateLimit` por Upstash Redis (`@upstash/ratelimit`) mantendo a mesma assinatura — os call sites não mudam.

## P6 · `select('*')` no config do WhatsApp por envio

**Onde:** `flows/meta-send.ts:80-84` (e equivalente em automations). Puxa a linha inteira (tokens criptografados inclusos) quando só precisa de 3 colunas. Micro, mas é caminho quente: `select('phone_number_id, access_token, user_id')`.

# PARTE 4 — EVOLUÇÃO DO MOTOR DE IA 💡

> O motor atual é sólido como *respondedor*: RAG híbrido, memória supervisionada, skills, contexto CRM, handoff, cap atômico. O que o separa de um **agente** de verdade é: ele não age, não vê mídia, não aprende com correções e responde a cada mensagem isolada. As melhorias abaixo estão em ordem de impacto.

## A · Tool use — deixar o agente AGIR no CRM (a maior evolução)

**Hoje:** uma única chamada de texto → texto. O modelo só sabe "responder ou desistir".

**Proposta:** dar ferramentas ao modelo via tool use nativo dos providers (ambos suportam):

| Ferramenta | O que faz | Por que importa |
|---|---|---|
| `buscar_conhecimento(query)` | 2ª rodada de RAG com query reformulada pelo modelo | O RAG atual usa a última mensagem literal como query; o modelo sabe reformular ("qnt custa aquele q vc falou" → "preço plano premium") |
| `criar_negocio(titulo, valor?)` | Insere deal no funil | Cliente diz "quero fechar" às 23h → lead entra no pipeline na hora, não se perde |
| `agendar_followup(quando, nota)` | Cria automação `scheduled` one-shot | "me chama semana que vem" vira ação real em vez de texto perdido |
| `atualizar_contato(campo, valor)` | Preenche email/empresa/custom fields | O agente coleta dados de qualificação sem formulário |
| `transferir_para_humano(motivo, urgencia)` | Handoff estruturado | Substitui o sentinel `[[HANDOFF]]` frágil (ver item D) por dados acionáveis |

**Implementação:** estender `generate.ts` para o loop de tool use (máx. 3 iterações), com cada ferramenta implementada server-side com o `accountId` **injetado pelo servidor** (o modelo nunca escolhe o tenant). Começar só com `buscar_conhecimento` + `transferir_para_humano` já muda o patamar.

**Por que é importante:** é a diferença entre um bot de FAQ e um SDR digital. Todo o valor de CRM do produto (pipeline, tags, campos) hoje é invisível para o agente — ele lê contexto mas não escreve nada.

## B · Enxergar mensagens não-texto

**Hoje:** `context.ts:28` filtra `content_type = 'text'`. Se o cliente manda um áudio ou foto do produto e pergunta "tem desse?", **o agente nem sabe que algo foi enviado** e responde no vácuo — a falha mais visível para o usuário final.

**Proposta em 2 fases:**
1. *Barato (1h de trabalho):* incluir placeholders no transcript — `[cliente enviou uma imagem: "legenda"]`, `[cliente enviou um áudio]` — mudando o select para incluir `content_type` e `media_url IS NOT NULL`. O prompt já instrui a encaminhar quando falta informação; com o placeholder ele encaminha *sabendo por quê*.
2. *Completo:* para imagens, baixar a mídia (o webhook já tem a URL) e passar como bloco de imagem para modelos com visão (Claude/GPT-4o); para áudio, transcrever via Whisper com a mesma chave OpenAI da conta.

## C · Debounce de rajadas de mensagens

**Hoje:** cliente digita "oi" / "tudo bem?" / "queria saber do plano" em 3 mensagens → 3 execuções completas do pipeline, 3 respostas do bot (parecendo metralhadora), 3 slots do cap consumidos.

**Proposta:** ao invés de responder no `after()` imediatamente, gravar um "pending reply" com `debounce_until = now() + 20s` na conversa (colunas novas ou tabela `ai_pending_replies`); o cron (ou um retrigger no próprio webhook: cada nova mensagem empurra o `debounce_until`) processa quando o cliente para de digitar, com o burst inteiro no contexto. **Por que:** é como um humano atende; corta custo de LLM em conversas reais em ~40-60% e elimina respostas fragmentadas.

## D · Handoff que avisa a equipe

**Hoje:** `auto-reply.ts:113-121` — no handoff, o bot desliga o auto-reply e... nada. O cliente fica sem resposta até alguém *casualmente* abrir o inbox. O sistema **já tem** tabela de notificações (migration 027) e presença de agentes (024).

**Proposta:** ao detectar handoff, (1) inserir notificação para os membros da conta com o motivo, (2) opcionalmente enviar ao cliente uma mensagem de transição configurável ("vou te passar para um colega 👍"), (3) marcar a conversa com tag/flag "aguardando humano" para filtro no inbox. Com a ferramenta `transferir_para_humano` do item A, a notificação carrega motivo e urgência estruturados.

## E · Observabilidade: registrar cada geração

**Hoje:** nenhum registro. Não dá para responder "quanto o agente custou esse mês?", "qual a taxa de handoff?", "as respostas pioraram depois que troquei o prompt?".

**Proposta:** tabela `ai_generation_logs` (account_id, conversation_id, mode draft/auto/playground, model, tokens in/out — os providers retornam `usage`, hoje descartado —, latency_ms, outcome sent/handoff/error, error_code) escrita fire-and-forget após cada `generateReply`. A UI de Intelligence (`ai-intelligence-panel.tsx`) ganha métricas reais. **Por que:** sem medição não existe melhoria de inteligência — é o pré-requisito de todos os outros itens.

## F · Aprendizado com correções humanas (o mais "parecido comigo")

**Hoje:** quando um humano **edita** o draft da IA antes de enviar, ou responde diferente do que o bot sugeriu, essa informação — o sinal de treino mais valioso que existe — é jogada fora.

**Proposta:** no send do composer, se havia draft de IA, comparar draft × texto enviado; se diferem significativamente, gravar o par em `ai_feedback` (pending). Um job (ou o próprio cron de consolidação) transforma padrões recorrentes em: (a) candidatos a memória/skill pendentes de aprovação, (b) exemplos few-shot para `conversation_examples`. **Por que:** cria o ciclo virtuoso agente→humano→agente; o sistema já tem a infra de aprovação supervisionada (memória pendente) — é estender o mesmo padrão para estilo e conteúdo.

## G · Retrieval mais inteligente

1. **Query de retrieval multi-turno:** `latestUserMessage` usa só a última mensagem — "e o preço?" não recupera nada. Concatenar as últimas 2-3 mensagens do cliente (`query.ts`, mudança de 5 linhas).
2. **FTS em português:** `to_tsvector('simple', ...)` não faz stemming — "preço" não casa "preços". Adicionar coluna gerada `fts_pt` com config `'portuguese'` e usar nas RPCs quando o locale da conta for pt-BR (mantendo `simple` como fallback), ou apostar de vez no caminho semântico.
3. **Skills com matching melhor:** `skills.ts:55` usa `query.includes(trigger)` — substring pura gera falsos positivos ("promo" dispara em "compromisso") e falsos negativos (plural). Mínimo: word-boundary regex (`\b`); ideal: incluir descrições das skills no mesmo índice FTS da memória.
4. **Embeddings para memória:** a memória é só lexical hoje; reutilizar a infra de `ai_knowledge_chunks` (vector 1536 + HNSW) para `ai_agent_memory` quando houver embeddings key.

## H · Saída estruturada na extração de memória

**Onde:** `memory-extract.ts:34-62` — parsing por regex `/\{[\s\S]*\}/` do texto livre do modelo. Funciona até o modelo comentar antes do JSON ou aninhar chaves. **Proposta:** usar structured outputs (OpenAI `response_format: json_schema` / Anthropic tool forçado) — ambos garantem JSON válido no schema. Elimina a classe inteira de falha silenciosa (`return []`).

## I · Prompt caching (custo e latência)

O system prompt (persona + exemplos + scaffold + skills) é longo e estável por conta; hoje é reprocessado a cada mensagem. Com Anthropic, marcar o bloco estável com `cache_control: {type: 'ephemeral'}` (ordenando: partes estáveis primeiro, KB/CRM/memória por último) reduz o custo de input em ~90% e o TTFT em conversas ativas. Exige mudar `system` de string para array de blocos no provider Anthropic — mudança pequena em `providers/anthropic.ts` + `defaults.ts` retornando `{stable, volatile}`.

## J · Humanização: typing, read receipts e delay natural

A Meta API suporta `mark_as_read` e typing indicator. Marcar como lida ao começar a gerar e simular digitação proporcional ao tamanho da resposta (cap ~5s) faz o atendimento parecer humano — e dá tempo do debounce (item C) agregar a próxima mensagem. Barato e de alto impacto percebido.

## K · Contexto por tokens + resumo rolante de conversa

`AI_CONTEXT_MESSAGE_LIMIT = 20` mensagens ignora o tamanho real; 20 mensagens longas estouram custo, 20 curtas desperdiçam contexto. Medir por caracteres/tokens estimados. Para conversas longas, manter um **resumo rolante** por conversa (coluna `ai_conversation_summary`, atualizado pelo cron de consolidação): o agente "lembra" que o cliente reclamou de entrega em abril mesmo fora da janela — memória episódica de longo prazo, complementar à memória de fatos.

## L · Locale por conta

`AI_PROMPT_LOCALE` é env global (`prompt-scaffold.ts:4`) — num deploy multi-tenant, todas as contas dividem o mesmo idioma de scaffold. Mover para coluna `ai_configs.prompt_locale` com fallback no env.

## M · Modelos: manter a lista viva

Além do fix B1: preferir sempre **aliases** (`claude-haiku-4-5`, `claude-sonnet-4-6`) a IDs datados — aposentadorias deixam de quebrar contas silenciosamente; e o `validateAiCredentials` já protege o save, mas vale rodar a validação também numa troca de modelo sem troca de chave (hoje roda — ok) e exibir aviso na UI quando o modelo salvo não está mais na lista curada.

## N · Playground com avaliação

O playground já espelha o caminho real. Evolução barata: botão "salvar como caso de teste" (pergunta + resposta esperada) e um runner que re-executa os casos após qualquer mudança de prompt/KB/skill, mostrando diff. **Por que:** é o que impede a regressão clássica "mexi no prompt e piorou tudo sem perceber" — versão mini de eval contínuo.
